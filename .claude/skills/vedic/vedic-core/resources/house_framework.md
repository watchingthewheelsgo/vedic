# 宫位诊断框架

> 用途：core在Step 3宫位诊断时参考
> 每个宫位执行：管理者审计 + 租客审计 + 相位审计 + 硬件带宽(SAV条件展开) + 分盘交叉（如有）

---

## 诊断四维度

### 1. 管理者（宫头星/宫主）
```
查看宫主的：
  P1身份 → 它是什么角色？
  去向(落宫) → 它去了哪个宫位？把这个宫的能量带到了哪里？
  P7尊贵度 → 它健不健康？
  P9 Shadbala → 它有没有力量执行？
```

### 2. 租客（宫内行星）
```
查看宫内星的：
  自带属性 → 吉星进驻=加持，凶星进驻=干扰
  合相情况 → 多颗星同宫=事务繁忙
```

### 3. 相位（谁在看这个宫位）
```
查看哪些行星对此宫位有相位（P10）：
  吉星相位 = 外部加持（保护/增益）
  凶星相位 = 外部压力（冲突/限制/放大）
  多星相位 = 该领域受到多方面外部影响
```

### 4. 硬件带宽（SAV）— 辅助维度
```
SAV是辅助确认指标，始终在数据表中展示，四档参考：
  SAV > 32 = 溢出（资源充裕）
  SAV 26-32 = 平稳（正常运作）
  SAV 20-25 = 紧张（需要额外努力）
  SAV < 20 = 匮乏（结构性不足）

特殊：凶宫(6/8/12) SAV低 = 反而是防火墙（贼进空屋）
      凶宫(6/8/12) SAV高 = 反转效果（破坏资源充足）
```

## 宫位评级

```
创始人模式：宫主强+宫内有吉星+吉星相位 → 自主掌控
经理人模式：宫主中等+正常相位 → 正常运作
吉祥物模式：宫主弱+有吉星相位 → 有贵人帮但自己弱
飘萍模式：宫主弱+无吉星+凶星相位 → 该领域缺乏根基
（SAV极端值可作为模式判定的辅助确认，但不作为主要判据）
```

## 事件关联陈述模板

在分析重点宫位时，自然嵌入Dasha事件关联：

```
"你的[X]宫主是[行星]，力量指数[Y]%([强/中/弱])，
 落在[Z]宫（[领域]）。

 在[大运行星]大运期间（[年份]-[年份]），
 你的[X宫领域]应该经历了[描述]...

 [如果宫主强]：这段时期你在这个领域有明确的进展和掌控感。

```

> 注意：这是陈述不是提问。用户不需要回答。
> 目的：展示分析实力 + 让用户在阅读中自行验证。

## 多阶段事件推导

大事件不是单一时间点，而是分阶段展开，每个阶段由不同宫主的Dasha触发：

| 事件 | 阶段1 (启动) | 阶段2 (落地) | 阶段3 (收益) |
|------|-------------|-------------|-------------|
| 结婚 | L7 (关系确立) | L9/9宫星 (法律确认) | L11 (社会公开) |
| 换工作 | L10 (机会/决定) | L6 (入职/竞争) | L11 (收入到位) |
| 买房 | L4 (动念/看房) | L2 (出钱/签约) | L4+D4 (入住) |
| 出国 | L9/L12 (决定) | L12 (离开) | L4 (安家) |

### 婚姻时间推导（三阶段模型）

现代婚姻是过程不是瞬间，Dasha映射三个阶段：

```
阶段1 — 关系确立/承诺：L7宫主的MD/AD
阶段2 — 法律确认(领证)：9宫内行星或L9宫主的MD/AD
阶段3 — 社会公开(婚礼)：11宫相关行星的MD/AD

⚠️ Venus不是婚姻首选触发器，除非Venus恰好是L7
⚠️ 三个阶段可能在同一个Antardasha内，也可能跨2-3个
⚠️ 输出格式："关系确立窗口在[L7期间]，正式落地在[9宫期间]"
   不要说"YYYY年结婚"这种单点预测
```

## Dasha事件推导硬约束（必须严格执行）

### 大运正负面判断规则

大运星的正负面不是"感觉"，是有明确判定条件的：

#### 正面条件（满足≥2条→判定为正面）
```
1. P1身份=吉角色（Core-Driver / Yogakaraka / Faithful）
2. P7尊贵度≥友方（Friend/Own/Exalted）
3. P9 Shadbala≥120%
4. 落在吉路（1/4/5/7/9/10宫）
5. 管的宫位SAV≥28
6. Vargottama或Dig Bala加成
```

#### 负面条件（满足任一即标注风险）
```
1. P1身份=凶角色（Destroyer管8/12 或 Growth-Hacker管6）
2. P2受损（燃烧/落陷/败相）
3. 落在凶路（6/8/12宫）且无VRY补救
4. P9 Shadbala<100%
5. 管的宫位SAV<25
6. 与凶星紧密合相（<5°）
```

#### 判定逻辑
```
正面≥2 且 负面=0 → 正面期（该领域顺利/上升）
正面≥2 且 负面≥1 → 混合期（发展中伴随压力）
正面<2 且 负面≥1 → 困难期（该领域面临挑战）
正面=0 且 负面≥2 → 危机期（需谨慎/有风险）
```

### 小运叠加规则
```
大运定基调，小运定具体事件。
小运星管哪个宫→那个领域在该时段被具体激活。

大运正 + 小运正 = 该领域顺利
大运正 + 小运负 = 好事多磨
大运负 + 小运正 = 困境中有转机
大运负 + 小运负 = 该领域面临危机（必须明确警示）
```

### ⚠️ 禁止的推导错误

```
1. 大运星管8宫+落陷 → 禁止说"深度转化带来成长"
   → 必须说"这段时期有突变/危机的风险"
   
2. 大运星管6宫+Shadbala弱 → 禁止说"在竞争中锤炼自己"
   → 必须说"可能面临健康或竞争方面的压力"
   
3. Venus大运+Venus入庙 → 不能因为知道用户那段时间不好就弱化
   → 必须说"这段时期在Venus管的领域有正面能量"
   
4. 凶宫主大运 → 不能美化成"成长的功课"
   → 凶就是凶，可以说"代价"但不能说"礼物"
   
5. 吉宫主大运 → 不能因为用户描述了困难就判负面
   → 吉星大运的正面能量是客观存在的，困难可能来自小运或流年
```

### Dasha事件优先级（多事件同期激活时的排序规则）

当用户问"先发生A还是先发生B"时（如"先换工作还是先结婚"），必须用以下规则排序：

```
规则1：Antardasha时序优先
  谁的Antardasha先到，谁的事件先发生。
  例：L10的AD在2026.3-2027.1，L7的AD在2027.1-2027.8
  → 换工作(10宫)先于结婚(7宫)

规则2：同一Antardasha内，大运主管宫优先
  如果两个事件在同一个Antardasha内都被激活：
  大运主(MD lord)管的宫位事务 > 小运主(AD lord)管的宫位事务
  → 大运定基调，与大运主相关的事件先落地

规则3：行星力量排序
  同时激活的两个事件，力量更强的行星（Shadbala更高）先显化。
  强星的事务更容易"冒出来"，弱星的事务需要更多外部催化。

规则4：多阶段事件的阶段分辨
  参照上方"多阶段事件推导"表：
  很多时候A和B并非互斥——可能A处于阶段1（启动），B处于阶段2（落地）。
  必须分辨用户问的是"启动"还是"完成"。
  例："换工作"可能阶段1(机会出现)在2026，阶段2(正式入职)在2027。

输出格式：
  "根据Dasha时间线：
   [事件A]的激活窗口在[AD期间]（[年月]-[年月]）
   [事件B]的激活窗口在[AD期间]（[年月]-[年月]）
   时间上[事件A]先于[事件B]约[X]个月。
   但[事件B]的启动信号(阶段1)可能与[事件A]的落地(阶段2)重叠。"
```

⚠️ **禁止**：不能用"感觉"或"一般来说"排序。必须查Dasha时间表给出具体Antardasha期间。

## 预测时间距离置信度（必须执行，覆盖career/love/core所有具体年份预测）

```
达萨盘算本身是精确的数学结果，但"某个大运期间会发生什么具体事件"这类解读的
经验校准度，会随预测窗口距离当前日期（structured_data生成时的current_date）
拉远而自然下降。距离越远，措辞必须越概括，禁止用同样的确定性语气处理近期和远期预测。

分级规则（按预测窗口起点距当前日期计算）：
  ≤3年（近期）  → 可给具体年月窗口 + 具体行星/宫位证据，正常颗粒度
  3-10年（中期） → 给年份区间（不细到月），措辞加"如果盘面结构不变"类限定语
  10-20年（远期）→ 只给"哪个大运/哪个人生阶段"级别的趋势描述，不给具体年份区间
  >20年（超远期）→ 只说"该大运整体基调是[正面/混合/困难]"，禁止描述具体事件内容

⚠️ 这条规则与盘主当前年龄是复合关系，不是互斥关系：
  例：2岁小孩的"事业定型期"预测在30年后 → 属于超远期，只能说"木星大运(2048-2064)
      整体基调正面，具体会落在哪个行业现在无法判断，需要成年后结合实际发展重新校准"
  例：30岁成年人问未来2年 → 属于近期，正常给出具体窗口和证据

禁止表达（远期预测中）：
  ❌ "2048年3月你会升任管理层"（超远期给出月份级具体事件）
  ✅ "2048年前后开始的木星大运，整体是事业上升期的基调，但具体形态要等实际
      成长路径明朗后才能细化"

这条规则不改变Dasha正负面判定本身（仍按house_framework"大运正负面判断规则"执行），
只改变"结论的颗粒度"——判定可以是确定的，但描述必须与预测距离匹配。
```

