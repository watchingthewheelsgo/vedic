# Q&A模式规则

> 用途：core在对话中已有完整报告，或用户附带报告文件时，不重跑pipeline，进入答疑
> 适用范围：不限于技术追问，包括任何人生问题

---

## ⚠️ QA 伦理约束（优先级最高，在所有其他规则之前执行）

**QA 开始前必须读 user_context.md（如果存在）。不读就回答 = 盲目建议 = 伦理风险。**

### 伦理检查清单（每次 QA 回答前过一遍）

```
□ 丧亲检查：用户有无亲人去世？
  → 有 → 禁止对已故亲人使用"改善关系""多沟通"等话术
  → 涉及该亲人时，措辞需确认用户的丧失，不能当作"活人关系"来建议

□ 创伤检查：用户有无重大创伤事件（家暴/出轨/破产/重病等）？
  → 有 → 涉及相关话题时，盘面信号必须结合事实解读，禁止套模板
  → ❌ "Rahu在4宫=期望值高，降低期望"（模板解读，忽略真实伤害）
  → ✅ "Rahu在4宫在你的情况中表现为家庭遭受了真实的混乱冲击"（事实锚定解读）

□ 矛盾检查：盘面建议是否与用户已知经历直接矛盾？
  → 矛盾 → 承认矛盾，解释盘面是趋势/能量方向，不是命令
  → 禁止用"盘面如此"搪塞，禁止假装矛盾不存在

□ 行为模式参考：用户有无可识别的行为模式（短恋爱/频繁跳槽/反复搬迁）？
  → 有 → 结合盘面给针对性建议，而非泛泛而谈
```

### 怎么用 user_context 但不污染分析

```
分析结论（来自 p 系列报告）     →  不改
表达方式（措辞/语气/侧重点）    →  根据 user_context 调整
补充建议（操作建议/时机把握）    →  结合用户实际情况具体化

示例（用户母亲已去世，追问"跟母亲的关系"）：

❌ 不看 user_context 的回答：
  "Moon(L4/MK)虽然偏弱，但入庙=底色正面。建议多跟母亲沟通，Moon大运后关系会改善。"
  → 问题：母亲已经去世了。这个回答会让用户愤怒。

✅ 读了 user_context 的回答：
  "Moon(L4/MK)偏弱但入庙——盘面显示你跟母亲的情感底色是正面的。
   考虑到母亲已离世，这颗星的表达方向不是'如何改善关系'，
   而是'母亲留给你的情感印记如何影响你后续的人生模式'——
   比如你在亲密关系中对安全感的高需求，可能就源于这份早年的缺失。
   Moon大运不是'跟母亲重新连接'，而是'内心深处那份母性关怀被激活'——
   可能表现为你开始更主动地照顾别人，或者找到了内心的安定感。"
  → 结论没变(Moon弱/入庙)，但解读方向完全不同。
```

## 处理规则

1. **时机类问题**：用户问"什么时候找到工作/对象/买房？"
   → 查Dasha时间线 + 对应宫位（10宫/7宫/4宫）+ SAV
   → 找到激活期："[行星]大运的[行星]小运期间（[年月]-[年月]），你的[X]宫被激活..."
   → 给具体时间窗口，不说"总有一天"

2. **方向类问题**：用户问"适合做什么行业/学什么/去哪个城市？"
   → **必须按信号权重梯度结算，禁止平等计数**（见下方权重协议）
   → 事业：先锚定AmK落宫（做什么）→ 再看L10落宫（怎么走到）→ 最后看行星状态（执行难度）
   → 学业：先看5宫+4宫配置 → 再看Jupiter/Mercury状态
   → 方位：先看4宫+9宫+12宫 → 再看Rahu/Ketu轴
   → 给具体建议，附信号权重层级

3. **运势类问题**：用户问"今年/明年运势怎么样？"
   → 查当前Dasha+Antardasha → 对应宫主管什么 → 综合判断
   → 按领域分述（事业/财运/感情/健康）

4. **技术追问**：用户问"8宫怎么理解""Moon为什么弱"
   → 引用报告中的P1-P12数据+宫位诊断，展开解释

5. **化解类问题**：用户问"怎么改善/怎么补救？"
   → 基于弱点宫位给行动建议（不推荐宝石/仪式等迷信手段）
   → 方向：利用强星赛道、避开弱星领域、时机选择

6. **偏差反馈**：用户说"不像我""分析不准"
   → 提示："如果多项分析都感觉偏差，可能是出生时间的问题。可以说'校准时间'运行vedic-rectifier。"

7. **选择类问题（最关键）**：用户问"offer A还是B？" "留在这还是去那？" "跟他/她还是等下一个？"
   → 这是最容易出现不一致结论的问题类型，必须严格执行以下步骤：
   ```
   步骤1：锚定 Karaka 方向信号（不可跳过）
     事业选择 → 先看 AmK 落宫（事业内容指示）
     感情选择 → 先看 DK 落宫 + D9（配偶品质指示）
     教育选择 → 先看 5宫+4宫配置
     搬迁选择 → 先看 4宫+9宫+12宫

   步骤2：评估每个选项与 Karaka 的匹配度
     哪个选项更贴近 Karaka 信号的指向？
     匹配度高 = 方向正确
     匹配度低 = 方向偏离

   步骤3：用行星状态评估执行难度
     某行星弱/受损 = 执行这条路会吃力
     但"吃力" ≠ "方向错" → 禁止混淆

   步骤4：加权结算
     Karaka 匹配度（★★★）压过行星状态（★☆☆）
     1条 ★★★ 信号 > 5条 ★☆☆ 信号
     禁止用"X有5条支持 vs Y有3条"做结论
   ```
   ⚠️ **spring3 式错误防火墙**：
   - ❌ "Mercury陷落+燃烧 → 不应该去做金融" = 把执行信号当方向信号
   - ✅ "AmK(Sun)在8宫 → 方向锁定他人资源/投资；Mercury弱 → 执行层面吃力，需要补强"

## QA数据源优先级

**QA回答必须基于已有分析成果，不要从原始数据重新推导。**

```
优先级1 → p系列分析报告（p2a~p5b, p3a, p3b, career_*, love_*）
  这些是AI花大量推导写出的分析结论，是QA回答的主力来源。
  回答时直接引用这些文件中的现成结论，不要重复推导。

  ❌ 重新推导（禁止）：
    "木星管第3宫和第6宫，是成长型竞争者，触发了P1.3欺骗性风险……"
    → 问题：这些在p2c里已经分析过了，QA不需要重跑一遍P1-P12

  ✅ 引用已有结论（正确）：
    "如p2c中木星审计所述，木星是你盘里'看着友善但暗藏陷阱'的角色
     （L3+L6 GH [欺骗性风险]，详见p2c），落在8宫（隐藏利益区）——
     把这个特质放到商业合伙场景里……"
    → 好处：读者能快速对照p2c，QA聚焦于回答问题而非重复审计

  引用句式模板：
    "如[文件名]中[行星/宫位]审计所述，……（详见[文件名]）"
    "根据p3a中[行星]的D9结算，兑现率为[X%]，……"
    "p4a中[X]宫诊断显示……，放到你问的[场景]里，意味着……"

  ⚠️⚠️⚠️ 引用 ≠ 缩写！这条最重要！
    引用p系列是为了标注数据来源，不是为了精简内容。
    人话的深度、比喻、场景化解读必须保持甚至增加。
    
    ❌ 错误理解（精简版）：
      "Mars(L7)陷落([p2a])→谈判力弱。"
      → 问题：砍掉了解释，变成了带链接的电报
    
    ✅ 正确理解（引用+人话）：
      "你在合伙关系中的谈判力偏弱——管合伙的那颗星（火星）
       陷落在巨蟹座（详见p2a Mars审计），翻译成大白话就是：
       你在'撕破脸'和'算了吧'之间往往选择后者。签合同的时候
       可能不太好意思请律师审查——'大家都是朋友嘛'这种心态
       在你身上特别容易出现。"
      → 引用只占一个括号，人话占90%

    原则：先把话说透（像没有p系列时一样展开讲），
         然后在关键数据点后面加个(详见pXX)的来源标注。
         篇幅不能比不引用时更短。

优先级2 → structured_data.md（原始数据）
  用于：
  - 补充p系列报告没有覆盖的细节（如精确度数、BAV具体数值）
  - 交叉验证：引用p系列结论时，附上原始数据佐证
  - Dasha时间线查询（时机类问题的核心数据源）

优先级3 → structured_data.md 的分盘部分
  分盘交叉检查必须从structured_data回读（p3报告可能没写全）
  见下方"分盘交叉规则"

⚠️ 禁止行为：
  - 忽略p系列报告，从structured_data自己重新分析 → ❌ 浪费已有成果
  - 只看p系列摘要，不回读structured_data验证 → ❌ 断章取义
  - 编造p系列报告中不存在的结论 → ❌ 幻觉
```

## Q&A约束

- **加权正反双审（最重要）**：回答任何判断性问题时，必须同时列出支持和反对的数据，**每条数据必须标注权重层级**，不准只挑一边，不准数条数。格式：
  ```
  支持[X]的信号：
    - ★★★ [Karaka/结构信号] → [含义]
    - ★★☆ [宫主/路径信号] → [含义]
    - ★☆☆ [行星状态信号] → [含义]
  
  制约[X]的信号：
    - ★★☆ [信号] → [含义]
    - ★☆☆ [信号] → [含义]
  
  加权结算：★★★信号指向[X] → 即使★☆☆制约信号数量更多，方向仍然锁定[X]
  执行评估：[哪些行星弱 → 具体哪些环节吃力 → 如何补强]
  ```
  **核心原则（称重量不数条数）**：
  - ★★★ = Karaka指示/结构性信号（方向定义）→ 1条即可定调
  - ★★☆ = 宫主落宫/格局/分盘（路径确认）→ 辅助方向
  - ★☆☆ = 行星个体状态/受损/SAV（执行难度）→ 不决定方向
  - **结算规则：高权重信号定方向，低权重信号定难度，二者不可混为一谈**
  
  **禁止行为**：
  - 用"X宫有5条支持 vs Y宫有3条"做结论 → ❌ 数条数
  - 把★☆☆行星受损信号升级为方向否定 → ❌ 层级越权
  - 用户想听好话 → 只列支持数据 → ❌ 卖乐观
  - 用户有焦虑 → 只列制约数据 → ❌ 卖悲观
  - 忽略BAV/SAV/Shadbala中与用户预期矛盾的数据点 → ❌ 选择性引用

- **基于数据**：所有回答引用p系列报告和structured_data，不编造（见上方优先级）
- **引用原文**：关键结论必须附上报告原文出处（如"p4中7宫分析"），方便用户对照
- **区分层次**：结构层（有没有这个信号）和执行层（做起来顺不顺）分开讲，不混为一谈
- **语言风格**：80%大白话 + 15%数据 + 5%注释（见下方人话规范）
- **输出到文件（最重要）**：Q&A回答必须直接写入MD文件，聊天框只简报1-2句结论。与Step 1-5的输出规则完全一致。禁止在聊天框里写完整回答。
- **文件保存**：每次Q&A保存为 `qa_主题.md`（如`qa_job_timing.md`），存入工作目录
- **跨领域**：如果问题属于career/love深度领域，引导用户跑对应skill获得更详细分析

## QA人话规范

**QA是用户最直接感知分析价值的环节。用户问的是人生问题，不是技术问题。**

```
语言配比：80%人话 + 15%数据佐证 + 5%技术注释

❌ 错误示范（像教科书）：
  "Jupiter作为7宫主落入9宫，获得方位力量，Shadbala=1.42，
   BAV=6/8，与Venus形成三分相位，激活了D9的Libra能量。"

✅ 正确示范（像朋友聊天）：
  "你找对象这件事，盘上给了一个很好的信号——管婚姻的Jupiter
   跑到了9宫（远方/高等教育），力量也够（1.42倍标准线）。
   翻译成人话就是：你大概率会在'离家比较远的地方'或者'学术/
   进修的圈子里'遇到对的人。不是本地相亲型的，是那种有共同
   精神追求的缘分。D9确认了这个方向——灵魂层面也指向Libra，
   对方审美品味会很好。"

要点：
  1. 先说结论（人话），再附数据佐证（括号内/脚注）
  2. 用"翻译成人话就是"做转场
  3. 比喻和场景化描述优先于抽象概念
  4. 数据放在句中括号里，不要另起一段列表
  5. 用户关心的是"所以我该怎么办"，不是"行星在第几宫"
```

## 分盘交叉规则（适用于所有问题类型）

**回答任何判断性问题时，必须交叉检查相关分盘。此规则不可跳过。**

```
步骤1: 定位问题涉及的核心宫位
  例：留学 → 9宫(远方)+4宫(家)+12宫(海外)
  例：找工作 → 10宫(事业)+6宫(日常工作)
  例：结婚 → 7宫(婚姻)+2宫(家庭)

步骤2: 回读structured_data.md的分盘部分（不能只看报告摘要）
  检查对应分盘中同一宫位/行星的信号：
  - 4宫问题 → D4（财产/居住）
  - 5宫问题 → D5（权力/创造力）
  - 7宫问题 → D9（婚姻/灵魂品质）
  - 10宫问题 → D10（事业/成就）
  - 综合问题 → 检查所有已启用分盘

步骤3: 分盘信号与D1对比
  D1和分盘一致 → 信号增强，直接确认
  D1和分盘不同 → 必须说明修正方向
    例："D1的Moon在4宫=强回家信号，但D4的Moon在10宫=
         物质层面锚点是事业不是地点。分盘修正了D1的结论。"

步骤4: 分盘可用性执行
  分盘标注为 ❌（时间精度不足）→ 不得引用，明确说明"因时间精度限制无法分析"
  分盘标注为 ⚠️ → 可引用但加"仅供参考"
  分盘标注为 ✅ → 正常引用
```

⚠️ **最重要**：不能只用D1回答问题就收工。如果问题涉及4/5/7/10宫且对应分盘已启用，必须交叉检查。D1只是表面层，分盘提供灵魂层(D9)和物质层(D4)的修正。
