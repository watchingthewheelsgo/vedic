# 关系性质盲扫 (signal-triage)

> 合盘 Layer 0.5。**不预设关系类型**，扫双方最响的跨盘信号，判定这段关系最突出的连接维度。
> 体系：KN Rao / Parashari，判据沿用 aspect-policy.md（整宫落点 + Graha Drishti + 度数强弱）。
> 数据来源：双方 structured_data.md 的 D1 行星位置 + Lagna + Moon/Venus/Mars + Rahu-Ketu + Dasha。

---

## 0. 目的

回答"这两个人之间最突出的连接在哪个维度"，而不是"他们恋爱合不合"。
输出**关系性质指纹** + 与用户声明诉求的**吻合度**，再据此进入对应关系框架（Layer 0-3）。

⚠️ 这一步不下承载/吉凶结论，只做性质归类。承载判断留给后续层。

---

## 0.5 语气铁规（最高优先级，最容易踩坑）

平扫阶段【关系类型尚未确定】，语言必须**中性**——只说"连接有多强、什么方向、什么性质"，不替关系下定义：

- ❌ 禁用关系类型专属词（此时还没确定是情侣 / 合作 / 朋友）：
  - 恋爱专属：勾住 / 上瘾 / 性张力 / 婚姻 / 暧昧 / 心动 / 配偶
  - 事业专属：KPI / 晋升 / 分红 / 客户
- ✅ 用中性词：吸引力强 / 能量互相激活 / 拉扯 / 约束 / 放大 / 摩擦 / 默契 / 牵引
- 同一信号在不同关系里含义不同——平扫只报"强度 + 方向"，把"具体是什么意思"留到进入框架后（Layer 2）。
  例：Venus-Mars 紧密 → 平扫说"强吸引、能量互相激活"，**不说**"性张力"（那是 romantic 框架才下的解读）。
- ⚠️ **中性 ≠ 平淡**：该强的信号要客观体现强度——用**专业、平稳但有分量**的措辞（全盘最强的接触 / 罕见的紧密合相 / 影响面广 / 多点位视相）配硬数据（如 orb `0°05'`）。语言科学克制：不夸张不口语（避免"最炸""拉满""火力"这类），也不堆参数读成体检报告。

## 2. 性质指纹归类

按"哪几类信号最响"归到一个或多个桶（可并列，标主次）：

| 性质指纹（中性，不预判好坏）| 主导信号组合 |
|---------|-------------|
| 强吸引（亲密拉力）| Moon-Venus / Venus-Mars 紧密、5/7宫互落、Moon-Moon 和谐 |
| 务实协作（事业咬合）| 10/11/2宫互落、Saturn·Mercury·Jupiter 建设性接触、6/10宫激活 |
| 业力/强迫性连接 | Rahu-Ketu 轴触 Moon/Venus/Lagna、Saturn 重压、8/12宫互落 |
| 平稳低戏剧 | 强接触少、吉星温和接触、无 Rahu/Mars 激化 |
| 高摩擦/高消耗 | 凶星紧密且修复资源缺、6/8/12 多重落点、Mars-Mars 对冲 |
| 亲缘/照护牵引 | Sun/Moon 跨盘紧密、4/9/5宫互落、Jupiter 庇佑（多见于家人/亲子）|

## 3. 输出格式（写入 reports/00_signal_triage.md）

```
关系性质指纹：[主性质] (+ [次性质])
最响的 3-5 条跨盘信号（方向句 + aspect-policy 判据 + 度数标注）
与用户声明诉求的吻合度：[高度吻合 / 部分吻合 / 诉求与盘面错位]
  → 错位时如实说明：用户想要 X，但两盘最强连接在 Y
进入框架：[romantic / business / friendship / family] 深析（Layer 0-3）
```

⚠️ 若盘面最强性质与用户诉求错位（如用户想谈恋爱、但两盘最响是事业互补 + 情感平淡），
必须**如实指出**，不顺着用户诉求把弱信号说成强信号（反确认偏误）。
